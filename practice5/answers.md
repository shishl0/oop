3. **Why was this "error" generated at run-time (rather than at compile-time)?**

4. **Add a try-catch statement. Re-compile `Example1`. What error is generated and why?**

In Java, any statement inside a `try` block is presumed to potentially fail via an exception. Hence, the compiler flags that execution might jump straight to the `catch` block (skipping the assignment line), leaving `ratio` uninitialized when you attempt to print it.

## Part 2: A More Complicated Example (`Example2.java`)

1. **Compile `Example2`. What error was generated?**  
   > `variable i might not have been initialized`. (In the line `numbers[i] + "/" + numbers[i + 1]`).

2. **Initialize `i` to 0 inside of the try block... What error was generated?**  
   > The same error: `variable i might not have been initialized`.

3. **It is not possible for `i` to be used before it is initialized. Why is this error generated anyway?**  
   Because Java's compiling rules dictate that *any* code inside a `try` block might throw an exception before finishing. Even though practically `i = 0` is simple and shouldn't throw an error, the compiler safely assumes that execution could unexpectedly drop into the `catch` block *before* the variable assignment completes. Thus, accessing `i` in the `catch` block is seen as unsafe.

4. **Move the initialization of `i` before the try block. Compile and execute... What output is generated?**
   ```text
   100/10=10
   Couldn't calculate 10/0
   ```

5. **Why aren't all of the divisions even attempted?**  
   Because the moment `10/0` triggers an exception, the control flow immediately leaves the `try` block (breaking out of the `for` loop in the process) and jumps to the single `catch` block at the bottom, successfully avoiding the rest of the elements in the array.

6. **Fix `Example2` so that it executes properly... What did you change? What has happened?**  
   - **Fix made:** I moved the `try-catch` wrapper so that it sits *inside* the `for` loop, rather than wrapping the `for` loop. 
   - **What happened:** This causes the try-catch sequence to trigger on a *per-iteration* basis. If one specific division fails (e.g. `10/0`), the loop simply hits the catch block, logs that specific failure, and proceeds back to the top of the loop for the next elements.

## Part 4 & 5: Some Other Exceptions and Programming Practice (`Example4.java`)

1. **What functionality does a `StringTokenizer` object provide? Give an example.**  
   A `StringTokenizer` provides the ability to break apart a large string into bite-sized sequential pieces (called "tokens") divided by specific separator characters.
   
   **Example:** 
   ```java
   StringTokenizer st = new StringTokenizer("apple,banana,pear", ",");
   ```
   *This would break the string into 3 tokens using commas.*

2. **What are the three formal parameters of the explicit value constructor in the `StringTokenizer` class? Give example.**
   - `String str`: The actual string content you want to break apart.
   - `String delim`: A string representing the delimiter characters that act as boundaries.
   - `boolean returnDelims`: A boolean determining if those delimiters themselves should *also* be returned as their own structural tokens.
   
   **Example:** 
   ```java
   new StringTokenizer("1+2", "+", true);
   ```

3. **Run `Example4`. Enter `5.3+9.2`. What output is generated?**
   ```text
   Result: 14.5
   ```

4. **Run again and enter `5.3+`. What output is generated?**
   ```text
   Invalid syntax
   ```

5. **Why? In particular, what exception is thrown and why?**  
   A `NoSuchElementException` is thrown. The `StringTokenizer` attempts to grab three specific tokens: the left operand, the operator, and the right operand. Since `5.3+` only possesses two tokens (evaluating down to `"5.3"` and `"+"`), the call to extract `rightString` fails because the tokenizer hit the end of the string.

6. **Run again and enter `5.3+a`. What output is generated?**
   ```text
   One or more operands is not a number
   ```

7. **Why? In particular, what exception is thrown and why?**  
   A `NumberFormatException` is thrown. The logic reads `"a"` successfully as a third token, but `Double.parseDouble("a")` strictly expects mathematical text formatting. It can't map `"a"` to a rational number, resulting in a parsing exception.
